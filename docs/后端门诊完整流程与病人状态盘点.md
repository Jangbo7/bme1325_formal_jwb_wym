# 后端门诊完整流程与病人状态盘点

> 审核基线：2026-06-18  
> 目的：盘点当前后端门诊流程、病人可能状态，以及这些状态在“就诊是否真正结束”上的语义，供后续前端 rule / 房间映射层对齐使用。

## 1. 当前状态语义的 source of truth

当前门诊状态不要只看单一字段，实际有四层语义：

1. `StandardOutpatientState`
   - 标准门诊编排状态机
   - 决定 encounter 在门诊流程里的正规迁移链路
2. `VisitLifecycleState`
   - 访问记录上的业务状态
   - 是后端对“当前处于哪一阶段”的主要持久化表达
3. `PatientLifecycleState`
   - 面向病人整体运行态的状态
   - 现在由 orchestration 层集中同步，不再在诊室 round2 / 无检查结束时过早写成完成
4. runtime projection
   - 由运行时投影层派生 `display_stage`、`dispatch_state`
   - 这是前端渲染和调度的直接消费面，不应反向定义后端业务真值

结论：后续映射层要以后端状态机和 visit/patient lifecycle 为主，`display_stage` / `dispatch_state` 只是投影。

## 2. 现在已经成立的门诊结束语义

当前后端已把“门诊结束”和“流程中间节点”重新拆开。

真正视为门诊完成的 visit state 只有：

- `in_emergency`
- `in_icu_rescue`
- `disposition_referral`
- `admitted`
- `transferring`
- `completed`

其中：

- 普通病人最终正常结束，应落到 `completed`
- 特殊去向病人，一旦最终去向明确，门诊阶段即结束
- `error` / `cancelled` 会停止自动推进，但不计入 `outpatient_flow_finished`

也就是说，下列状态都不再算 finished：

- `waiting_payment`
- `medical_payment_completed`
- `disposition_pending`
- `waiting_pharmacy`
- `disposition_outpatient_treatment`
- `disposition_followup_booking`

## 3. 普通病人的当前完整流程

### 3.1 普通病人：有药 / 需药房

这是现在的标准普通闭环：

```text
DIAGNOSIS_FINALIZED
-> WAITING_MEDICAL_PAYMENT
-> MEDICAL_PAYMENT_COMPLETED
-> DISPOSITION_PENDING
-> DISPOSITION_PHARMACY
-> COMPLETED
```

对应事件链：

```text
request_medical_payment
-> pay_medical
-> plan_disposition
-> choose_pharmacy
-> dispense_medication
```

关键约束：

- pharmacy 已进入主流程，不再是流程外附加动作
- `finish/completed` 必须位于 pharmacy 之后
- `dispense_medication` 已成为正确的药房完成事件

### 3.2 普通病人：无药 / 不走药房

当前普通无药闭环：

```text
DIAGNOSIS_FINALIZED
-> DISPOSITION_PENDING
-> DISPOSITION_OUTPATIENT_TREATMENT 或 DISPOSITION_FOLLOWUP_BOOKING
-> COMPLETED
```

对应事件链：

```text
plan_disposition
-> choose_outpatient_treatment / choose_followup_booking
-> complete_visit
```

关键约束：

- `followup_booking` 现在已正确映射到 `choose_followup_booking`
- `disposition_outpatient_treatment` / `disposition_followup_booking` 本身不再等于 finished
- 必须再经过 `complete_visit` 才算门诊结束

## 4. 特殊去向病人的结束语义

以下去向一旦确定，即视为门诊阶段结束：

- `in_emergency`
- `in_icu_rescue`
- `disposition_referral`
- `admitted`
- `transferring`

这类病人不是“普通完成出院”，但对门诊链路来说已经结束，因此会进入 finished 语义。

## 5. error / cancelled 的处理

当前语义是：

- `error`
- `cancelled`

这两类状态：

- 会停止自动化推进
- 会在 runtime projection 中进入 error fallback 渲染语义
- 但不会被计入 `outpatient_flow_finished`

因此它们不是 finished lane，而是异常/保底 lane。

## 6. PatientLifecycleState 的当前约束

这次改动后，病人生命周期的完成判定被后移并集中化：

- 普通病人不再在诊室 round2 或无检查收尾时就提前写成 completed
- 只有当 visit 进入真正 finished 的状态，patient lifecycle 才会由 orchestration 层统一同步为 completed
- `error` -> patient lifecycle `error`
- `cancelled` -> patient lifecycle `cancelled`

这解决了“后端病人状态已经 finish，但实际还没缴费/没去药房”的提前完成问题。

## 7. 自动规划器与直接 API 的差异

当前要区分两条消费路径：

### 7.1 自动规划器 / runner

自动 runner 会继续推进完整链路，包括：

- 医疗缴费
- disposition 计算
- 药房发药
- 普通门诊 `complete_visit`

也就是说，自动病人现在可以自然跑完整闭环。

### 7.2 直接 API 调用方

如果外部只调用到诊断收尾或中间节点，后续事件仍需要显式驱动，例如：

- `pay_medical`
- `plan_disposition`
- `choose_pharmacy`
- `dispense_medication`
- `complete_visit`

否则 encounter 会停留在中间状态，这属于当前设计预期，不是 bug。

## 8. 当前病人可能状态盘点

下面按“是否仍在普通门诊主流程中”分类。

### 8.1 仍在普通门诊主流程中的状态

| visit state | 含义 | 是否 finished |
|---|---|---|
| `arrived` / `triaged` / `registered` 等前置状态 | 进入门诊前半段 | 否 |
| `waiting_call` | 等叫号 | 否 |
| `in_consultation` / `in_second_consultation` | 诊室中 | 否 |
| `test_ordered` / `waiting_test_payment` / `test_payment_completed` / `in_exam` / `waiting_test_results` / `results_ready` | 检查链路中 | 否 |
| `diagnosis_finalized` | 诊断已收束，但未完成后续闭环 | 否 |
| `waiting_payment` | 等医疗缴费 | 否 |
| `medical_payment_completed` | 医疗费已缴，但尚未落最终 disposition | 否 |
| `disposition_pending` | 已进入去向判断 | 否 |
| `waiting_pharmacy` | 等药房发药 | 否 |
| `disposition_outpatient_treatment` | 普通门诊治疗/观察类结果，尚未 complete_visit | 否 |
| `disposition_followup_booking` | 随访预约类结果，尚未 complete_visit | 否 |

### 8.2 已结束门诊阶段的状态

| visit state | 含义 | 是否 finished |
|---|---|---|
| `completed` | 普通门诊完整结束 | 是 |
| `disposition_referral` | 转诊去向已确定 | 是 |
| `admitted` | 已收住院 | 是 |
| `transferring` | 已进入转运链路 | 是 |
| `in_emergency` | 已转急诊 | 是 |
| `in_icu_rescue` | 已转 ICU/抢救 | 是 |

### 8.3 异常/停止状态

| visit state | 含义 | 是否 finished |
|---|---|---|
| `cancelled` | 已取消 | 否 |
| `error` | 进入错误态 | 否 |

## 9. 对映射层设计的直接影响

后续前端映射层需要遵守以下约束：

1. 不要把 `waiting_payment`、`medical_payment_completed`、`disposition_pending`、`waiting_pharmacy` 渲染成 finished
2. 不要把 `disposition_outpatient_treatment`、`disposition_followup_booking` 直接渲染成完整结束
3. `error` / `cancelled` 要进入异常保底渲染，而不是 finished
4. 普通病人的 finish 必须在药房后或 `complete_visit` 后出现
5. 特殊去向病人的 finish 语义来源于“门诊结束”，不是“已离院”

## 11. 结论

当前后端门诊主流程已经具备以下特征：

- pharmacy 已真正进入普通病人主流程
- 普通病人的 finish 已后移到 pharmacy 之后或 `complete_visit` 之后
- 特殊去向病人在去向确定时结束门诊阶段
- `error` / `cancelled` 保留为异常态，不冒充 finished
- patient lifecycle 的 completed 不再提前写入

基于这个基线，后续前端 rule / 房间映射层可以开始做“状态投影对齐”，而不是继续围绕旧的提前 finished 语义补洞。
