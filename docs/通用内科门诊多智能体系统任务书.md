# 通用内科门诊多智能体系统任务书

## 一、项目背景
当前项目已经具备一个可运行的医院场景原型，拥有以下基础能力：
- 前端 3D/伪 3D 医院场景与角色移动
- 分诊台交互与分诊卡录入
- 分诊对话页与多轮问答
- 后端 `FastAPI + LangGraph + State Machine + EventBus + Repository` 的基础框架
- 分诊完成后的排队联动与状态持久化

这说明项目已经不再是单点原型，而是具备了进一步扩展为完整医疗流程模拟系统的技术基础。

结合当前系统现状，下一阶段最重要、最直观、最适合对外展示的目标，不是继续横向增加零散功能，而是先把一条**患者视角的完整门诊流程**做出来。相比 ICU 场景，门诊场景的优势在于：
- 流程更清晰，更适合当前前后端结构承载
- 玩家体验路径完整，更适合演示和验收
- 多角色协作自然存在，但不会像 ICU 一样高度依赖高频监护数据
- 便于后续逐步扩展到多科室、多 Agent、多分支流程

参考 ICU 任务书时，本项目借鉴的是以下思路：
- 分层架构（流程层、Agent 协作层、支撑层）
- 事件驱动（Event-driven）
- 分层记忆（Shared Memory / Agent Private Memory / Working State）
- 多角色协作与可解释输出

本项目**不直接复刻 ICU 的业务目标**，因为 ICU 任务书核心是高频监护、风险预警、时序数据驱动，而当前项目更适合先完成一个以**流程驱动 + 对话驱动 + 场景节点驱动**为核心的门诊系统。

技术实现与协作方式详见 [通用内科门诊多智能体系统开发手册](./通用内科门诊多智能体系统开发手册.md)。

## 二、项目总目标
本项目的 V1 总目标是：

**构建一个基于当前代码框架的“患者视角通用内科门诊完整流程 Demo”，让玩家能够从进入医院开始，经过挂号、分诊、候诊、面诊、检验/缴费/药房等必要环节，最终完成一次完整门诊并离院。**

该目标强调三件事：
1. **流程完整**：不是只做分诊，也不是只做对话，而是做一条能闭环的门诊链路
2. **角色清晰**：核心角色有明确职责，核心智能由关键 Agent 承担
3. **可持续扩展**：V1 先收敛到通用内科门诊，后续再扩展到多科室、多角色、多复杂场景

V1 的最重要验收标准是：

**玩家能够稳定、完整、可复现地跑通一次门诊流程。**

这里的“稳定”和“完整”包括：
- 前后端状态一致
- 对话与排队可以联动
- 服务节点顺序合法
- 存在标准开药路径和检验回诊路径两种主要流程

## 三、系统范围

## 3.1 In Scope
V1 明确包含以下内容：

### 业务范围
- 通用内科门诊
- 患者视角的单次门诊完整流程
- 挂号、分诊、候诊、面诊、缴费、检验、药房、离院

### 核心智能角色
- `分诊护士 agent`
- `门诊医生 agent`

### 流程节点角色
- 挂号窗口
- 收费窗口
- 检验窗口
- 药房窗口

### 环境支持角色
- 其他患者
- 导诊 / 保安

### 技术范围
- 在现有 `FastAPI + triage agent + State Machine + EventBus + Repository + scene 前端模块` 基础上演进
- 增加门诊主流程实体和状态控制
- 保留现有分诊、排队、前端场景等已有成果

## 3.2 Out of Scope
V1 明确不做以下内容：
- 多科室并行门诊
- ICU 高频监护系统
- 住院、手术、急诊抢救链路
- 所有角色都做成真 Agent
- 复杂医嘱系统、真实 EHR 系统、真实收费系统
- 研究型多中心数据分析或医疗预测模型比较实验

V1 的基本策略是：
**先完成门诊闭环，再扩展场景复杂度。**

## 四、角色与系统组成

## 4.1 玩家角色
玩家扮演**患者**。玩家需要：
- 进入医院
- 在不同节点按流程行动
- 与护士和医生进行对话
- 在服务窗口完成挂号、缴费、检验、取药等步骤
- 最终完成就诊流程并离院

## 4.2 核心角色

### 分诊护士 Agent
职责：
- 负责门诊入口的初步问诊
- 采集症状、起病时间、疼痛、体温、过敏信息等
- 做初步风险判断和科室建议
- 决定患者是否进入普通内科候诊
- 输出可解释的分诊结论与追问结果

说明：
- 现有 `triage agent` 将被正式收编为“分诊护士 agent”
- 它是 V1 的第一个核心真 Agent

### 门诊医生 Agent
职责：
- 负责门诊面诊阶段的多轮问答
- 补充现病史和必要信息
- 做简单评估与判断
- 决定是否直接开药/建议离院，或进入检验流程
- 在回诊阶段解释检验结果并给出处置建议

说明：
- 这是 V1 新增的第二个核心真 Agent
- 它是门诊闭环成立的关键角色

## 4.3 辅助流程角色
这些角色在 V1 中主要作为**流程节点**存在，而不是深度智能体：
- 挂号窗口
- 收费窗口
- 检验窗口
- 药房窗口

其职责是：
- 按当前门诊状态推动流程向前
- 展示当前节点信息
- 更新后端状态与结果
- 与场景位置绑定，增强真实感

## 4.4 环境角色
V1 中以下角色主要承担环境与引导职责：
- 其他患者
- 导诊
- 保安

其作用包括：
- 增强医院真实感
- 形成候诊和排队氛围
- 对玩家提供位置或流程提示
- 控制玩家进入未开放区域

V1 中这些角色**不作为主流程的关键决策点**。

## 五、核心流程
V1 至少要稳定支持两条完整路径。

## 5.1 标准开药路径
标准开药路径用于体现“普通门诊直接完成处置”的最基础闭环：

1. 玩家进入医院
2. 前往挂号窗口完成挂号
3. 前往分诊台完成分诊
4. 进入候诊区等待内科叫号
5. 进入诊室完成门诊医生面诊
6. 医生直接给出处置建议并开药
7. 玩家前往药房取药
8. 玩家完成门诊并离院

该路径的意义在于：
- 最短路径清晰
- 最适合作为第一次演示和第一轮验收主线
- 可以作为其他复杂路径的基础模板

## 5.2 检验回诊路径
检验回诊路径用于体现“门诊不是一次对话即结束，而是可进入次级流程并返回”的能力：

1. 玩家进入医院并挂号
2. 完成分诊并进入候诊
3. 医生首诊后判断需要做检验
4. 医生开具检验相关要求
5. 玩家前往收费窗口完成缴费
6. 玩家前往检验窗口完成检验
7. 检验结果回传系统
8. 玩家回到医生处完成回诊
9. 医生给出最终处置意见
10. 玩家前往药房取药，或直接离院

该路径的意义在于：
- 体现系统能处理“门诊中的中间分支”
- 让医生 Agent 有明确的二阶段职责
- 证明状态机与 EventBus 能支撑更复杂的流程跳转

## 六、分层设计
为了保证 V1 可持续扩展，系统采用分层设计。

## 6.1 患者流程层
这一层关注的是：
- 患者当前处于门诊的哪个阶段
- 下一个可进入的合法阶段是什么
- 哪些节点是必经，哪些节点是可选分支

这是整个系统的主线层，也是后续 `Visit Session` 和全局门诊状态机的归属层。

## 6.2 Agent 协作层
这一层包含：
- 分诊护士 Agent
- 门诊医生 Agent

其职责是：
- 通过多轮问答采集信息
- 基于共享患者信息和私有对话进度做判断
- 输出结构化结论与建议

这一层是系统的“核心智能部分”，但并不负责整个门诊流程推进的所有工作。

## 6.3 服务节点层
这一层包括：
- 挂号
- 缴费
- 检验
- 药房

这些节点的目标是：
- 让门诊成为真正的流程闭环
- 以可见的场景节点方式推进状态
- 保持与前端空间交互一致

## 6.4 环境角色层
这一层包含：
- 其他患者
- 导诊 / 保安

职责：
- 增强场景真实感
- 提供辅助提示
- 构成可观察的多角色环境

V1 中它们主要服务于“门诊体验”，而不是承担复杂业务逻辑。

## 6.5 前后端支撑层
这层是整个系统运行的基础设施，包括：
- 前端场景模块
- 后端 API
- Agent 运行时
- State Machine
- EventBus
- Repository
- 持久化数据库

该层不直接决定业务目标，但决定系统能否稳定承载门诊主线。

## 七、阶段目标

## Phase 1：门诊主流程骨架
目标：
- 建立门诊级别的主流程对象和顶层状态管理
- 明确从到院、挂号到候诊的基础链路
- 将当前 triage 能力正式纳入门诊主流程

结果：
- 系统不再只有 triage 主线，而是存在一次完整门诊的概念

## Phase 2：分诊护士 Agent 与候诊联动
目标：
- 升级当前 triage agent 为“分诊护士 agent”
- 分诊完成后稳定进入候诊、排队、叫号逻辑
- 保证分诊结果与队列状态联动稳定

结果：
- 门诊入口阶段闭环成立

## Phase 3：门诊医生 Agent
目标：
- 新增医生面诊 Agent
- 支持门诊对话、历史采集、简单判断与两类处置分支
- 形成从候诊到面诊的核心就诊链路

结果：
- 真正的门诊“看医生”环节成立

## Phase 4：检验 / 缴费 / 药房闭环
目标：
- 完成辅助服务节点
- 支持直接开药路径与检验回诊路径
- 让一次门诊真正从入口到离院闭环

结果：
- V1 完整门诊 Demo 成立

## Phase 5：环境角色增强
目标：
- 丰富其他患者、导诊、保安等角色表现
- 增强医院氛围、秩序感和引导感

结果：
- 从“能跑通”升级到“更像真实医院”

## 八、验收标准
V1 的验收必须按**完整业务流程**来定义，而不是按零散功能点定义。

## 8.1 标准开药路径完成
必须能够完整跑通：
- 到院
- 挂号
- 分诊
- 候诊
- 面诊
- 药房
- 离院

要求：
- 无非法状态跳转
- 前后端显示一致
- 玩家能明确知道当前下一步要做什么

## 8.2 检验回诊路径完成
必须能够完整跑通：
- 首诊
- 开检验单
- 缴费
- 检验
- 回诊
- 最终处置

要求：
- 检验结果能返回并影响后续面诊
- 回诊不是重置流程，而是正常衔接门诊主线

## 8.3 状态流转合法
系统必须保证：
- 不可跳过挂号直接面诊
- 不可跳过分诊直接进入候诊
- 不可跳过缴费直接检验
- 不可在 visit 完成后非法重入旧状态

## 8.4 前后端一致
必须保证：
- 前端看到的患者阶段、对话、排队、节点状态与后端一致
- 玩家交互结果能稳定映射到后端流程状态
- 退出对话或再次回到节点时不会丢失流程上下文

## 8.5 对话与队列可稳定联动
必须保证：
- 分诊对话能够稳定驱动候诊/排队
- 医生面诊完成后能稳定驱动下一个服务节点
- 对话结果不是孤立文本，而是能推动门诊主流程向前

---

## 九、后续路线
V1 完成后，V2 / V3 可以扩展到以下方向：
- 多科室并行门诊
- 更多深度 Agent（如检验解释 Agent、药房说明 Agent）
- 更复杂的导诊、保安、其他患者行为
- 更复杂的医嘱与随访流程
- 更丰富的医院空间与节点系统
- 更强的可解释性、审计能力和演示视图

但后续路线的前提始终是：
**先把通用内科门诊 V1 做成稳定闭环。**
