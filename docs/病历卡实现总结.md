# 病历卡实现总结

**日期**：2026-05-22

## 1. 结论

当前项目里的“病历卡”并不是单独的一套病历系统，而是由两部分组成：

- **后端病历流水**：负责把就诊过程中的关键事件按时间线落库
- **前端病历卡视图**：负责把这些数据渲染成左下角的病历面板

也就是说，病历卡本质上是一个**面向场景展示的投影层**，它从后端就诊数据和病历流水中读取信息，再展示给玩家。

## 2. 后端病历结构

当前后端病历采用两张核心表：

- `medical_records`：一条 visit 对应一条总病历头记录
- `medical_record_entries`：按时间追加的病历条目

对应仓库实现见：

- [backend/app/repositories/medical_records.py](../backend/app/repositories/medical_records.py)
- [backend/app/database.py](../backend/app/database.py)

### 2.1 `medical_records`

这个表保存病历头信息，主要字段是：

- `id`
- `patient_id`
- `visit_id`
- `created_at`
- `updated_at`

它更像是“某次就诊的病历索引”。

### 2.2 `medical_record_entries`

这个表保存具体条目，字段包括：

- `phase`
- `entry_type`
- `actor`
- `title`
- `content_text`
- `content_json`
- `created_at`

它更像是“病程记录 / 检查记录 / 问诊记录”的流水。

## 3. 谁在写病历

当前主流程里，至少有两条清晰的写入来源：

### 3.1 分诊写入

分诊完成后，triage agent 会追加一条分诊摘要，通常包含：

- 主诉
- 分诊等级
- 优先级
- 科室
- 说明性 note

写入位置见：

- [backend/app/agents/triage/service.py](../backend/app/agents/triage/service.py)

### 3.2 内科写入

内科医生 agent 会把问诊和检查相关结果写入病历流水，尤其是：

- 辅助检查仿真报告
- 第二轮问诊和最终计划

写入位置见：

- [backend/app/agents/internal_medicine/service.py](../backend/app/agents/internal_medicine/service.py)

### 3.3 其他读取方

除了写入，部分调试 agent 和场景快照服务也会读取病历时间线：

- [backend/app/services/scene_snapshot_service.py](../backend/app/services/scene_snapshot_service.py)
- [backend/app/agents/patient_agent/controller.py](../backend/app/agents/patient_agent/controller.py)
- [backend/app/agents/npc_patient/controller.py](../backend/app/agents/npc_patient/controller.py)

## 4. 前端病历卡怎么展示

前端病历卡在 [scene/ui/medical-record.js](../scene/ui/medical-record.js) 中实现。

它的核心逻辑是：

1. 先同步患者和 visit 数据
2. 把不同来源的数据整理成四个展示区
3. 再用 Canvas 把这些信息画出来

### 4.1 展示的四个区域

前端当前会显示：

- **患者信息**：ID、姓名、年龄、性别
- **分诊信息**：分诊等级、科室、症状、体温、心率、血压
- **问诊信息**：医生、诊断、处方、备注
- **检查报告**：类别、项目、报告文本

### 4.2 数据来源

当前前端的数据来源是：

- 患者信息：`patient`
- 分诊信息：`patient.triage_data`
- 问诊信息：前端仍预留 `visit.data.internal_medicine_data` 的展示位，但后端主流程更稳定写入的是 `visit.data.diagnostic_session`、`visit.data.internal_medicine_round2_summary` 和 `visit.data.simulated_report`
- 检查报告：`visit.data.simulated_report`

### 4.3 现实状态

目前最稳定、最完整的是：

- **患者信息**
- **分诊信息**
- **检查报告**

而 **问诊信息** 这一块前端展示位还保留 `internal_medicine_data`，但后端主流程目前更常写入 `diagnostic_session`、`internal_medicine_round2_summary` 和 `simulated_report`，字段名还没有完全统一。

## 5. 对外接口

当前提供的病历读取接口是：

- `GET /api/v1/medical-records/visit/{visit_id}`

对应实现见：

- [backend/app/api/routes/medical_records.py](../backend/app/api/routes/medical_records.py)

这个接口返回的是该 visit 的完整病历时间线，适合：

- 前端调试页
- 场景快照服务
- 后续的病历详情页

## 6. 当前实现的特点

### 6.1 优点

- 病历不是简单拼字符串，而是有独立的时间线表结构
- 分诊、内科、检查结果都能进入同一条 visit 病历流
- 前端展示和后端存储解耦，便于后续扩展

### 6.2 限制

- 前端病历卡和后端字段还没有完全一一对应
- `internal_medicine_data` 目前在前端已有展示位，但后端主流程的稳定写入键主要是 `diagnostic_session`、`internal_medicine_round2_summary` 和 `simulated_report`
- 病历卡更像“运行态摘要面板”，还不是完整的 EMR 系统

## 7. 后续如果要继续完善，建议的方向

1. 统一 `visit.data` 里的病历字段命名
2. 统一 `visit.data` 里的病历字段命名，特别是 `internal_medicine_data` 与 `diagnostic_session` / `internal_medicine_round2_summary`
3. 给病历卡加一个更标准的时间线视图
4. 把检查结果、问诊结果、分诊结果都沉淀成同一套展示 schema
