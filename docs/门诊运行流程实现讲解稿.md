# 门诊运行流程实现讲解稿（技术评审短版）

## 1. 文档目的
这份文档用于对外解释本项目“门诊流程是如何被代码实现出来的”。

讲解对象：技术评审、开发者。

讲解目标：
- 让听众理解患者从到院到离院的流程闭环。
- 让听众看到流程不是写在文档里，而是由状态机、事件流、仓储层和 API 共同驱动。
- 明确当前已实现能力与后续扩展边界。

## 2. 一句话总览
系统把一次门诊拆成“可观测的状态迁移”，并通过事件将分诊、排队、患者展示等模块解耦联动，形成可复现的流程闭环。

---

## 3. 两条核心路径（业务视角）

### 3.1 标准开药路径
1. 到院并创建本次就诊 Visit。
2. 进入分诊对话，生成分诊建议。
3. 分诊完成后自动进入候诊队列。
4. 叫号后进入面诊。
5. 开药后去药房，完成离院。

### 3.2 检验回诊路径
1. 到院、分诊、候诊与标准路径一致。
2. 首诊后医生判断需要检验。
3. 进入缴费和检验环节。
4. 检验结果回传后进入回诊。
5. 最终处置后取药或离院。

说明：当前代码已经具备 Visit、分诊、队列联动的基础闭环；医生深度对话和检验节点细化属于下一阶段扩展。

---

## 4. 运行时如何推进流程（实现视角）
以下按一次分诊流程说明关键链路：

1. 前端发起分诊会话请求。
2. 后端分诊图流程执行：加载上下文 -> 评估 -> 持久化 -> 返回响应。
3. 服务层写入 shared/private memory、对话轮次、患者状态、会话状态。
4. 分诊完成时发布事件，队列订阅者创建排队号并推进患者状态。
5. 前端再次查询患者和队列数据，得到一致展示。

关键入口：
- 创建 Visit: [backend/app/api/routes/visits.py](../backend/app/api/routes/visits.py)
- 创建分诊会话与追问: [backend/app/api/routes/triage.py](../backend/app/api/routes/triage.py)
- 查询患者与队列: [backend/app/api/routes/patients.py](../backend/app/api/routes/patients.py), [backend/app/api/routes/queues.py](../backend/app/api/routes/queues.py)

## 5. 三层状态机协同

### 5.1 患者全局状态（PatientLifecycleState）
定义患者在门诊中的全局业务位置，例如 untriaged、triaging、queued、in_consultation、completed。

代码位置：
- 状态定义: [backend/app/schemas/common.py](../backend/app/schemas/common.py)
- 迁移规则: [backend/app/domain/patient/state_machine.py](../backend/app/domain/patient/state_machine.py)

### 5.2 单次就诊状态（VisitLifecycleState）
定义“一次门诊 Visit”中的阶段，例如 arrived、triaging、waiting_consultation、waiting_test、completed。

代码位置：
- 状态定义: [backend/app/schemas/common.py](../backend/app/schemas/common.py)
- 迁移规则: [backend/app/domain/visit/state_machine.py](../backend/app/domain/visit/state_machine.py)
- 持久化: [backend/app/repositories/visits.py](../backend/app/repositories/visits.py)

### 5.3 分诊对话状态（TriageDialogueState）
定义分诊会话内部的多轮追问过程，例如 idle、evaluating、needs_followup、awaiting_patient_reply、triaged。

代码位置：
- 状态机: [backend/app/agents/triage/state_machine.py](../backend/app/agents/triage/state_machine.py)
- 图编排: [backend/app/agents/triage/graph.py](../backend/app/agents/triage/graph.py)

结论：
- Patient/Visit 负责业务流程位置。
- TriageDialogue 负责分诊子流程。
- 三者解耦但联动，避免“一个状态机管全部”导致复杂度失控。

## 6. 分诊 Agent 的执行闭环

### 6.1 编排结构
分诊运行时由 service + graph 组合：
- graph 控制节点顺序。
- service 负责规则评估、LLM 调用、记忆写入、状态推进、事件发布。

代码位置：
- [backend/app/agents/triage/graph.py](../backend/app/agents/triage/graph.py)
- [backend/app/agents/triage/service.py](../backend/app/agents/triage/service.py)

### 6.2 评估与追问
service 会结合：
- 规则库检索与规则兜底。
- 已有 shared memory 与对话 turns。
- 缺失字段优先级策略。

代码位置：
- 规则与抽取: [backend/app/agents/triage/rules.py](../backend/app/agents/triage/rules.py)
- 规则数据: [backend/app/rag/rule_store.json](../backend/app/rag/rule_store.json)

### 6.3 持久化与响应
分诊每轮会更新：
- 会话轮次 session_turns。
- 患者 shared memory。
- agent private memory。
- 患者状态和 visit 状态。

并返回统一响应给前端。

## 7. 事件驱动联动（核心可解释点）

### 7.1 关键事件
- TRIAGE_COMPLETED
- PATIENT_STATE_CHANGED
- VISIT_STATE_CHANGED
- QUEUE_TICKET_CREATED

代码位置：
- 事件常量: [backend/app/events/types.py](../backend/app/events/types.py)
- 事件总线: [backend/app/events/bus.py](../backend/app/events/bus.py)

### 7.2 典型链路
分诊完成后：
1. 发布 TRIAGE_COMPLETED。
2. QueueSubscriber 监听并创建排队号。
3. 推进患者到 queued，并发布状态变化事件。
4. 前端读取队列接口看到新号码。

代码位置：
- 队列订阅: [backend/app/events/subscribers/queue.py](../backend/app/events/subscribers/queue.py)

设计收益：
- 分诊模块不直接依赖队列模块。
- 后续扩展 doctor、缴费、检验时可复用同样事件模式。

## 8. 数据与记忆分层

### 8.1 Shared Memory
存患者稳定事实（主诉、症状、体征、过敏等），可跨 Agent 读取。

### 8.2 Agent Private Memory
存某个 Agent 的私有过程状态（追问历史、缺失字段、快照等）。

### 8.3 Working State
图运行中的临时态，用于本轮计算，不直接作为长期事实。

代码位置：
- memory repository: [backend/app/repositories/agent_memory.py](../backend/app/repositories/agent_memory.py)
- triage working memory: [backend/app/agents/triage/schemas.py](../backend/app/agents/triage/schemas.py)

## 9. 为什么这个实现可扩展

1. 新 Agent 可以按统一目录模板接入，不破坏现有 triage。
2. 主流程由 Visit 状态机承载，后续节点可按状态和事件逐步增量接入。
3. Repository 层隔离了持久化细节，便于从 Demo 数据升级到更稳定存储。

可参考：
- [backend/app/agents/triage](../backend/app/agents/triage)
- [通用内科门诊多智能体系统开发手册](./通用内科门诊多智能体系统开发手册.md)

## 10. 当前实现边界（讲解时必须明确）
已具备：
- Visit 实体与状态机骨架。
- 分诊 Agent 多轮对话和结果结构化输出。
- 分诊完成后的自动入队联动。
- 患者/会话/队列/记忆持久化。

正在扩展：
- 医生 Agent 的完整对话子状态机与业务决策流。
- 检验、缴费、药房节点的细化规则与前端交互完整闭环。

这样表述可以保证评审对“已完成能力”和“下一阶段计划”有清晰预期。

## 11. 8-12 分钟讲解建议（可直接照此讲）

1. 第 1 分钟：讲目标。
2. 第 2-3 分钟：讲两条门诊路径。

4. 第 7-8 分钟：讲分诊完成如何自动入队（事件链路）。
5. 第 9-10 分钟：讲数据与记忆分层。
6. 第 11-12 分钟：讲可扩展性与当前边界。

当前系统已经从“单点分诊原型”演进为“可状态追踪、可事件联动、可持续扩展”的门诊流程平台。
