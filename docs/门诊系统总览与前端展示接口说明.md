# 门诊系统总览与前端展示接口说明

> 目标读者：
> 1. 需要快速理解整个门诊后端实现的人
> 2. 需要基于后端数据设计更好看 GUI 的前端合作者
> 3. 需要为项目评鉴展示准备演示链路的人

## 1. 这份文档解决什么问题

这份文档不是最低层细节文档，也不是一次性开发记录。它的作用是：

- 用一条完整患者链路解释整个门诊系统是怎么跑起来的
- 告诉前端同学“哪些字段是业务真值，哪些字段是为了展示准备的投影层”
- 告诉演示页面应该重点展示哪些内容，才能让评鉴者一眼看懂系统价值
- 把读者引导到更细的专题文档

如果只能先读一份文档，建议先读这一份。

本次展示 GUI 的定位也在这里明确一下：

- 它本质上不是 `scene/` 那种单患者交互页
- 它更接近一个更细、更好看、信息密度更高的 `department-runtime-debug`
- 它的核心任务是把“多患者如何在各科室里流转”展示清楚

## 2. 一句话理解整个系统

当前系统本质上是一个“患者在医院门诊里流转”的模拟引擎。

后端做四件核心事情：

1. 维护患者和就诊的真实业务状态
2. 驱动分诊、问诊、检查、缴费、处置、转诊等流程
3. 生成适合前端展示的投影数据
4. 把患者移动事件同步给更大的全院可视化系统（Fullview）

换句话说，前端最终看到的一切界面变化，背后都可以追溯到一条真实的后端状态链。

## 3. 从“一个患者”视角看完整门诊链路

### 3.1 到院与分诊

患者进入系统后，通常先经历：

- `arrived`
- `triaging`
- `waiting_followup`
- `triaged`

这一段由 `triage` agent 负责收集主诉、危险信号、建议科室和优先级。

前端展示建议：

- 患者卡片上突出“当前阶段 = 分诊”
- 显示分诊等级、优先级、目标科室
- 若有追问，展示当前缺失字段和最新追问内容

对应源码：

- `backend/app/agents/triage/*`
- `backend/app/domain/visit/state_machine.py`
- `backend/app/schemas/common.py`

### 3.2 挂号与排队

分诊完成后，患者进入目标科室门诊链路：

- `registered`
- `waiting_consultation`
- `in_consultation`

这时后端已经开始把患者放进“某个科室、某个队列、某个房间入口”的语义里。

前端展示建议：

- 用“队列中 / 已叫号 / 进入诊室”三段式展示
- 显示目标科室、队列票据、医生位或诊室位
- 让评鉴者一眼看到系统不是简单聊天，而是真有门诊调度

如果展示主界面是 department dashboard，那么这一段更适合表现为：

- 患者进入某个 department gate
- 患者进入该科室等待区
- 患者被叫到某个 doctor slot 或 consultation room

对应源码：

- `backend/app/repositories/queues.py`
- `backend/app/services/department_runtime_service.py`
- `backend/app/services/runtime_projection.py`

### 3.3 首轮医生问诊

当前主线门诊医生有：

- `internal_medicine`
- `surgery`

首轮问诊阶段，后端会根据病人情况决定三类大方向：

1. 直接门诊完成
2. 进入检查或门诊处置
3. 升级去急诊 / ICU / 住院 / 转诊

前端展示建议：

- 对话区展示 agent 回复和患者回复
- 旁边展示结构化结果摘要，例如：
  - 当前科室
  - 轮次（round 1 / round 2）
  - 是否需要检查
  - 当前 disposition
- 不要只展示聊天气泡，要同步展示“系统判断”

对应源码：

- `backend/app/agents/internal_medicine/*`
- `backend/app/agents/surgery/*`
- `backend/app/services/disposition.py`

### 3.4 检查 / 门诊处置 / 回诊

如果医生认为需要进一步处理，患者可能走到：

- `waiting_test`
- `waiting_test_payment`
- `test_payment_completed`
- `in_test`
- `waiting_return_consultation`
- `results_ready`
- `waiting_second_consultation`
- `in_second_consultation`

或者：

- `waiting_outpatient_procedure`
- `in_outpatient_procedure`

这一段很适合做可视化，因为它最能体现“流程系统”的价值。

前端展示建议：

- 把首轮、检查、回诊做成时间线或泳道图
- 对检查结果用卡片形式展示
- 明确标出“患者现在是在等待、处理中、还是已出结果”

对于 department dashboard 来说，这一段尤其值得强调，因为它最能体现：

- 患者不是只在单个科室静止不动
- 患者会在“科室问诊位 / 检查位 / 支付位 / 回诊位”之间发生真实流转

对应源码：

- `backend/app/agents/test_simulator/service.py`
- `backend/app/services/encounter_orchestration.py`
- `backend/app/services/runtime_projection.py`

### 3.5 缴费、取药、转诊、急诊升级、住院

问诊完成后，后端并不会简单把患者全部归成“结束”，而是继续细分：

- `diagnosis_finalized`
- `waiting_payment`
- `medical_payment_completed`
- `disposition_pending`
- `waiting_pharmacy`
- `disposition_referral`
- `admitted`
- `transferring`
- `completed`
- `in_emergency`
- `in_icu_rescue`

这正是展示时最容易讲清楚系统深度的部分。

前端展示建议：

- 把“门诊结束”拆成真正的去向，而不是统一用一个 finished 图标
- 对转诊 / 急诊 / ICU / 住院使用明显不同的颜色和路径
- 在评鉴演示时，强调系统不仅能聊天，还能做完整去向管理

对应源码：

- `backend/app/services/disposition.py`
- `backend/app/schemas/orchestration.py`
- `backend/app/domain/visit/state_machine.py`

## 4. 前端必须理解的五层状态

这一部分最重要。前端如果不区分这五层，界面很容易“看起来能跑，但语义是错的”。

### 4.1 `VisitLifecycleState`

这是 visit 当前真实流程阶段。

它回答的问题是：

- 患者这一次门诊目前走到哪一步了？

详细说明：

- [outpatient_states.md](./outpatient_states.md)

### 4.2 `PatientLifecycleState`

这是病人当前运行态，更偏系统调度语义。

它回答的问题是：

- 这个病人当前是否在分诊、排队、诊中、检查中、已完成？

### 4.3 `StandardOutpatientState`

这是标准门诊状态机的统一业务层抽象。

它回答的问题是：

- 当前 encounter 在门诊标准流程里属于哪一类业务节点？

### 4.4 `disposition`

这是医生或系统给出的结构化去向。

它回答的问题是：

- 这个患者下一步应该去哪？

例如：

- `outpatient_treatment`
- `followup_booking`
- `specialty_referral`
- `emergency_escalation`
- `icu_rescue`
- `inpatient_admission`

### 4.5 `runtime projection`

这是专门给前端和调试页消费的展示层。

它回答的问题是：

- 前端应该把当前患者渲染成“分诊 / 等待 / 诊中 / 检查 / 缴费 / 完成”中的哪种展示状态？

核心字段：

- `display_stage`
- `dispatch_state`
- `consultation_round`
- `blocking`
- `resource_assignment`

详细说明：

- [CURRENT_ARCHITECTURE_VISUALIZATION.md](./CURRENT_ARCHITECTURE_VISUALIZATION.md)
- [门诊后端行为状态到前端Rule与房间映射.md](./门诊后端行为状态到前端Rule与房间映射.md)

## 5. 前端展示时最值得强调的字段

如果目的是做“给评鉴者看的系统 GUI”，而且主界面本质上是一个增强版 `department-runtime-debug`，建议优先把下面这些字段做成显眼信息。

### 5.1 患者主卡片

每个患者主卡片建议至少展示：

- `patient_id`
- `visit_id` / `encounter_id`
- `assigned_department_name`
- `visit_state`
- `patient_lifecycle_state`
- `display_stage`
- `dispatch_state`
- `consultation_round`
- `primary_disposition`
- `outpatient_flow_finished`

这样评鉴者能同时看到：

- 业务真值
- 展示状态
- 去向判断

### 5.2 资源位置

以下字段非常适合做空间可视化：

- `resource_assignment.department_gate_name`
- `resource_assignment.doctor_slot_name`
- `resource_assignment.consultation_room_name`
- `resource_assignment.current_node_id`
- `resource_assignment.target_node_id`
- `resource_assignment.target_resource_kind`

如果要做“患者移动动画”或“当前位于哪个房间”的 GUI，这组字段是第一优先级。

对 department dashboard 来说，最建议把它们画成：

- 科室入口
- 等候队列
- 医生位
- 检查位
- 缴费位
- 药房位
- 转运去向

### 5.3 阻塞与异常

以下字段适合做提示条、徽标或警告面板：

- `blocking.kind`
- `blocking.resource_kind`
- `blocking.resource_name`
- `blocking.message`
- `last_error`
- `latest_consultation_llm_error`

这能很好展示系统不只是 happy path，还能解释为什么病人卡在某一步。

### 5.4 检查与病历摘要

以下字段很适合做“详情侧栏”：

- `medical_record_summary`
- `latest_test_report`
- `current_dialogue_preview`
- `carry_forward_summary`
- `recommended_department_reason`

这部分展示出来，评鉴者会更容易理解“AI 对话”和“流程状态”之间不是脱节的。

## 6. 前端最适合对接的 API 分层

为了做一个更好看的 GUI，不建议所有页面都直接拼原始表数据。建议按展示目的分层调用。

### 6.1 单患者故事视图

最适合做“玩家视角 / 单病人主界面”的接口：

- `GET /api/v1/scene-snapshot`
- `GET /api/v1/patients/{patient_id}`
- `GET /api/v1/visits/{visit_id}`
- `GET /api/v1/medical-records/visit/{visit_id}`
- `GET /api/v1/queues`

其中：

- `scene-snapshot` 最适合直接驱动一个完整的病人主页面
- 它已经打包了：
  - `self_patient`
  - `active_visit`
  - `active_queue_ticket`
  - `active_dialogue`
  - `medical_record_summary`
  - `latest_test_report`
  - `ui_flags`
  - `other_patients`
  - `queues`

### 6.2 多患者调度总览

这部分是你这次展示 GUI 的主基底。最适合做“更细更美观的 department dashboard / 评鉴展示大屏”的接口：

- `GET /api/v1/runtime-console/snapshot`
- `GET /api/v1/runtime-console/patients`
- `GET /api/v1/runtime-console/departments`
- `GET /api/v1/runtime-console/events`
- `GET /api/v1/hospital-runtime/snapshot`
- `GET /api/v1/department-runtime-debug/snapshot`

适合展示：

- 当前活跃患者数
- 科室分布
- 智能病人 / 脚本病人比例
- 当前每个病人的 `display_stage`
- 当前每个病人的 `dispatch_state`
- 当前每个病人的 `department_status`
- 当前每个病人的 `resource_assignment`
- 阻塞和异常
- 各部门负载

如果要选一个“最像现有 prototype 的起点”，优先从：

- `GET /api/v1/department-runtime-debug/snapshot`

开始，因为它天然就是 department-centric 的数据视角。

### 6.3 Encounter / 状态机解释视图

如果要做“点击某患者后展开细节”的解释面板，建议接这些：

- `GET /api/v1/encounters/{encounter_id}`
- `GET /api/v1/encounters/{encounter_id}/state-debug`
- `GET /api/v1/state-machine/graph`

适合展示：

- 当前状态节点
- 下一步允许事件
- 标准门诊状态机图

这对评鉴者特别有用，因为它能清楚证明系统是“有状态机支撑”的。

### 6.4 Fullview 联动视图

如果 GUI 想展示“后端决定了移动，前端地图是否已经接收”，建议接：

- `GET /api/v1/fullview-sync/control`
- `GET /api/v1/fullview-sync/outbox`
- `GET /api/v1/fullview-sync/projection/{patient_id}/{encounter_id}`
- `GET /api/v1/fullview-sync/encounters/{encounter_id}`

适合展示：

- 当前 movement / transfer / discharge 请求是否发送成功
- 是否被 step gate 阻塞
- 是否已被观察到
- 当前 Fullview 投影位置

### 6.5 实时事件流

如果你们想做“页面自动跳动”的动态演示，可以考虑：

- `GET /api/v1/events/stream`

适合用来驱动：

- 时间线更新
- 状态徽标刷新
- 最近事件列表

## 7. 面向评鉴展示，GUI 最值得做的五个模块

### 7.1 Department Dashboard 主画面

这是最核心的画面，本质上就是“增强版 `department-runtime-debug`”。

建议按 department 分栏或按楼层/科室分区展示：

- 每个科室当前有多少患者
- 每个科室有哪些患者在等待、诊中、检查中、缴费中、转运中
- 每个 doctor slot / room 当前占用情况
- 每个患者下一跳目标位置

这一块应该是评鉴展示时最常驻的主屏。

### 7.2 患者旅程总览

建议做成一条可滚动时间线，包含：

- 到院
- 分诊
- 挂号
- 排队
- 首轮问诊
- 检查 / 处置
- 回诊
- 缴费 / 取药 / 转诊 / 急诊 / ICU / 住院

这是最能让非技术评鉴者理解系统价值的模块。

### 7.3 患者状态卡片墙

建议每个患者一张卡，突出：

- 科室
- 当前阶段
- 当前去向
- 是否阻塞
- 是否特殊事件

这能直接展示系统的并发处理能力。

### 7.4 科室与房间占用视图

建议显示：

- 每个部门当前多少患者
- 哪些医生位 / 房间被占用
- 哪些患者在等待叫号
- 哪些患者卡在资源容量或 Fullview gate

这部分最能体现“不是单一聊天机器人，而是完整门诊调度系统”。

如果 UI 精力有限，这块甚至可以直接和主 dashboard 合并成一个更强的 department-runtime 主视图。

### 7.5 单患者详情侧栏

建议包含：

- 最新对话
- latest test report
- medical record summary
- 当前状态链
- disposition
- 当前 room / node

适合在演示时点开一个患者，讲完整故事。

### 7.6 系统健康与异常面板

建议单独展示：

- 当前 issues
- blocking 原因
- Fullview sync backlog
- 最近 runtime events
- LLM unavailable / 配置异常

评鉴者通常会问“失败场景怎么看”，这一块最好提前准备。

## 8. 前端合作时的两个重要约定

### 8.1 展示字段不等于业务真值

前端可以优先消费：

- `display_stage`
- `dispatch_state`

但一旦涉及严格业务判断，必须回看：

- `visit_state`
- `disposition`
- `outpatient_flow_finished`

不要反过来让展示层字段定义业务语义。

### 8.2 不要只做静态原型

如果这套 GUI 最后要给评鉴者演示，建议至少保证下面三种变化能真实联动：

1. 患者状态切换
2. 队列 / 科室人数变化
3. Fullview 同步状态变化

这样展示效果会远比“只画一个看起来漂亮的静态页面”更有说服力。

## 10. 一句话给前端合作者

如果你要做的是“给评鉴者看的更好看 GUI”，而且核心是一个更强的 `department-runtime-debug`，最值得强调的不是把页面做成很多按钮，而是把下面三件事讲清楚：

- 患者在系统里到底走到了哪一步
- 系统为什么让他去那里
- 如果他卡住了，是被什么资源或规则卡住了

只要这三件事讲清楚，这个系统的门诊价值就会非常直观。
