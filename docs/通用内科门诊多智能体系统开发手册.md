# 通用内科门诊多智能体系统开发手册

## 一、手册目的
本手册用于指导项目组在**当前代码框架**基础上推进“通用内科门诊多智能体系统”的开发。

这份文档不是从零重做系统设计，也不是抽象愿景说明，而是回答以下问题：
- 当前已有的前后端框架能承接什么
- 下一步应该先做什么，后做什么
- 各模块应该如何分工
- State Machine、EventBus、记忆设计各自的边界是什么
- 新增 Agent 应该按什么方式接入，才能不破坏现有架构

产品目标与业务边界详见 [通用内科门诊多智能体系统任务书](./通用内科门诊多智能体系统任务书.md)。
架构全景参考详见 [CURRENT_ARCHITECTURE_VISUALIZATION.md](./CURRENT_ARCHITECTURE_VISUALIZATION.md)。

## 二、当前代码框架概览
当前系统已经具备一个相对清晰的模块化单体架构，可作为后续门诊扩展的基础。

## 2.1 后端结构
当前后端主结构为：
- `FastAPI` API 层
- `triage agent` 运行时
- 患者与分诊对话的 `State Machines`
- 进程内 `EventBus`
- `Repository` 持久化层
- 本地数据库存储 patient / session / memory / queue

关键分层如下：
- `backend/app/main.py`
  - 应用装配入口
- `backend/app/api/routes/`
  - REST API
- `backend/app/agents/triage/`
  - 当前唯一完整 Agent 包
- `backend/app/domain/patient/state_machine.py`
  - 患者生命周期状态机
- `backend/app/events/`
  - EventBus 与 subscriber
- `backend/app/repositories/`
  - 数据访问层
- `backend/app/schemas/`
  - Pydantic 契约

## 2.2 前端结构
当前前端采用原生场景化结构：
- `scene/main.js`
  - 入口文件
- `scene/core/bootstrap.js`
  - 前端 orchestration layer
- `scene/agent/`
  - agent client、对话与表单逻辑
- `scene/queue/`
  - queue runtime
- `scene/npc/`
  - NPC runtime
- `scene/ui/`
  - task board 等 UI 模块

## 2.3 当前可复用能力
当前框架里，最值得直接复用的能力有：
- 分诊对话与多轮问答
- 分诊结果结构化返回
- 分诊完成后的队列联动
- 前端场景内节点交互与 `E` 键触发模式
- 后端显式状态机与事件流
- 分层记忆模型的雏形

## 2.4 当前框架的关键判断
现有 `triage agent` 不是废弃物，而是下一阶段“分诊护士 agent”的直接基础。当前框架的正确演进方式不是推翻重来，而是：

**先在现有 triage / queue / scene / state-machine 框架上加一个更高层的门诊主流程，再逐步接入医生 Agent 和辅助节点。**

## 2.5 当前 V1 严格流程（已落地）
当前代码已按严格顺序推进：
- `triage_completed -> visit=triaged`
- `register_completed -> visit=registered`
- `queue_wait_elapsed(10s) -> visit=waiting_consultation`
- `start_consultation -> visit=in_consultation`

对应动作 API：
- `POST /api/v1/visits/{visit_id}/register`
- `POST /api/v1/visits/{visit_id}/progress`
- `POST /api/v1/visits/{visit_id}/enter-consultation`

前端交互点：
- 挂号点：`registration`（按 `E` 触发 register）
- 医生入口点：复用 `pharmacy` 位置（显示 `Doctor Entry`，按 `E` 触发 enter-consultation）

协作边界：
- 主流程与状态推进由 visit 路由/状态机维护。
- 医生 Agent 开发者仅需改 `backend/app/agents/outpatient_doctor/**`，不需要改 visit 主链路。

## 三、推荐开发主线
以下开发顺序固定，建议严格遵守。原因是每一步都在为下一步清障。

## 1. 先建 `Visit Session`
当前系统已经有：
- triage session
- patient state
- queue state

但还没有一个真正代表“一次完整门诊”的主实体。因此第一步必须先建立 `Visit Session`。

`Visit Session` 的作用：
- 作为一次门诊的顶层业务对象
- 统一挂接挂号、分诊、候诊、面诊、检验、药房等阶段
- 成为未来多 Agent 协作的统一根节点

如果跳过这一步，后面医生 Agent、检验、回诊都会继续依附在 triage session 上，结构会越来越乱。

## 2. 再把 triage 接入 visit 主线
第二步不是重写 triage，而是把现有 triage 能力接入新的门诊主流程。

目标：
- 让 triage 成为 `Visit Session` 的一个阶段
- 让现有分诊护士逻辑继续可用
- 让分诊结束后自然进入候诊/排队阶段

原因：
- 这样能最快把现有成果转化为门诊系统的一部分
- 也能为医生 Agent 提供入口前置条件

## 3. 再做挂号 / 候诊 / 叫号
门诊 Demo 的核心不是只有对话，还包括真实的空间节点和中间状态。

这一阶段要补的是：
- 挂号窗口节点
- 候诊状态
- 队列与叫号
- 从分诊结束到进入医生环节的完整过渡

原因：
- 如果不先补这一层，医生 Agent 会直接挂在 triage 后面，丢掉门诊流程感
- 队列是现有系统已经有基础能力的环节，优先把它扩到门诊主线最划算

## 4. 再做 doctor agent
医生 Agent 是 V1 的第二个真 Agent，但不能抢在主流程之前做。

推荐这样做的原因：
- 医生 Agent 一定依赖清晰的门诊阶段语义
- 需要明确它的输入来自哪里、结束后推进到哪里
- 需要和 visit、queue、memory、回诊节点协同

如果在 visit 主状态不清晰前就做医生 Agent，后面必然返工。

## 5. 再做检验 / 缴费 / 药房
这一阶段是为了补齐完整闭环。

实现方式建议是：
- 做成场景内可见服务节点
- 先做规则化状态推进
- 不急着把每个窗口都做成真 Agent

原因：
- V1 的关键目标是“完整门诊 Demo”，不是“每个角色都智能”
- 检验/缴费/药房先做成可靠流程节点，收益最高、风险最低

## 6. 最后做 supporting NPC
其他患者、导诊、保安等 supporting NPC 应该最后增强。

原因：
- 这些角色对沉浸感有帮助，但不是 V1 主流程成立的前提
- 太早投入会分散精力，导致主流程一直闭不上

一句话原则：
**先立主流程，再补核心智能，再补辅助节点，最后做环境增强。**

## 四、模块分工建议
当前项目适合按“能力层”拆任务，而不是按“一个角色一个人全包”来拆。

## 4.1 流程与状态机组
负责：
- `Visit Session`
- 门诊主生命周期状态机
- triage 对话状态机与门诊流程对接
- doctor consultation 子状态机
- 服务节点状态推进规则

目标：
- 确保所有关键状态和状态迁移是显式、可测试、可解释的

## 4.2 Agent Runtime 组
负责：
- triage agent 的重命名与升级
- doctor agent 新增
- prompts / rules / graph / state / service 的标准化维护
- Agent 私有记忆与 Shared Memory 的接入

目标：
- 保证真 Agent 的能力增强建立在统一结构之上

## 4.3 数据与事件组
负责：
- `Visit Session`、patient、session、queue、memory 仓库设计
- EventBus 事件定义
- subscriber 逻辑
- persistence schema
- audit 与 projection

目标：
- 保证状态变化、事件广播、数据持久化相互解耦

## 4.4 前端交互组
负责：
- 场景节点交互
- triage dialogue UI
- doctor dialogue UI
- task board
- queue board
- 场景引导提示

目标：
- 让玩家清楚知道自己在门诊主流程中的位置和下一步动作

## 4.5 环境角色组
负责：
- supporting NPC
- 候诊区氛围
- 导诊/保安提示
- 路线限制与非关键交互

目标：
- 在不干扰主流程的前提下提升沉浸感

## 五、状态机约定
当前系统已经有两类状态机，未来必须继续沿这个方向演进。

## 5.1 患者全局生命周期状态机
这是整个门诊流程的主状态机。

职责：
- 决定患者现在处于门诊的哪个阶段
- 决定哪些下一步是合法的
- 控制是否允许进入挂号、分诊、候诊、面诊、检验、药房等节点

这个状态机是全局流程裁判，不能被前端对话逻辑替代。

## 5.2 triage 对话状态机
这是现有分诊对话的子状态机。

职责：
- 管理分诊对话的多轮追问
- 判断是否缺字段
- 决定分诊对话何时完成

注意：
- 它只负责“分诊会话”的流程，不负责整次门诊的业务位置

## 5.3 doctor consultation 子状态机
这是未来新增医生 Agent 后必须拥有的子状态机。

职责：
- 管理首诊、补充追问、判断、开单、回诊等阶段
- 限制医生 Agent 的合法流程边界

## 5.4 状态机与 EventBus 的分工
这个边界必须非常明确：
- `State Machine` 负责定状态
- `EventBus` 负责广播事实
- `Subscriber` 或其他模块负责处理副作用

也就是说：
- 状态机先判断“能不能变、变到哪”
- 状态已经提交后，再发事件
- 不能让 EventBus 自己决定主流程往哪里走

这是当前框架最重要的技术红线之一。

## 六、记忆设计约定
门诊系统会逐渐出现多个 Agent，因此记忆设计必须严格分层。

## 6.1 `shared memory`
用于存放稳定、可复用、可审计的事实。

推荐内容：
- 患者基础资料
- 主诉
- 已确认症状
- 起病时间
- 生命体征
- 过敏史
- 检验结果
- 已确认的关键临床事实

规则：
- 只存“事实”
- 可被多个 Agent 读取
- 不能混入某个 Agent 的私有推理进度

## 6.2 `agent private memory`
用于存放某个 Agent 的私有进度和内部上下文。

例如：
- triage agent 的缺失字段、追问历史、recommendation snapshot
- doctor agent 的面诊阶段、已问项目、待确认项

规则：
- 某个 Agent 的 private memory 不能污染另一个 Agent
- private memory 与 `agent_type + session_id` 绑定

## 6.3 `working state`
这是 graph 运行时的临时态。

作用：
- 存放本轮调用中的中间上下文
- 保存暂时性的推测或节点计算结果

规则：
- 不直接当作长期业务事实存储
- 除非被确认，否则不要写回 shared memory

## 6.4 写入规则
这是门诊系统长期稳定的关键约定：
- 事实进 `shared memory`
- 过程进 `agent private memory`
- 推测留在 `working state`

如果不守这个边界，后续多 Agent 协作会很快失控。

## 七、新增 Agent 的标准方式
未来新增任何 Agent，都不能脱离当前框架乱接。

## 7.1 标准目录结构
新增 Agent 时，目录结构固定推荐为：

```text
backend/app/agents/<agent_name>/
  graph.py
  state.py
  state_machine.py
  schemas.py
  prompts.py
  rules.py
  service.py
```

## 7.2 各文件职责
- `graph.py`
  - 定义该 Agent 的 graph 流程
- `state.py`
  - 定义 graph 运行态
- `state_machine.py`
  - 定义该 Agent 的局部流程状态
- `schemas.py`
  - 该 Agent 的输入输出契约
- `prompts.py`
  - 纯 prompt 和文案模板
- `rules.py`
  - 规则化策略、解析逻辑、fallback
- `service.py`
  - 编排 memory、graph、state transition、repository、response

## 7.3 新增门诊医生 Agent 的接入要求
未来新增 doctor agent 时，必须遵守：
- 不能写回 `triage/` 包中
- 不能直接使用 triage private memory 作为自己的内部状态
- 必须挂在 `Visit Session` 的后续阶段中
- 必须使用独立 consultation 子状态机
- 必须通过 shared memory 继承 triage 已确认事实

## 八、前端开发约定

## 8.1 场景节点优先于纯面板跳转
当前项目的优势之一是有场景感，因此门诊扩展应优先保持“空间节点驱动”。

推荐：
- 玩家走到挂号窗口按 `E` 完成挂号
- 走到分诊台按 `E` 进入分诊
- 走到诊室门口按 `E` 进入医生对话
- 走到收费/检验/药房窗口按 `E` 推进流程

不推荐：
- 把整条门诊做成纯按钮点击流程

## 8.2 对话 UI 与流程状态分离
前端对话页只负责展示和发送消息，不负责决定门诊流程如何迁移。

也就是说：
- 前端显示当前阶段
- 前端发起动作
- 后端状态机决定是否允许进入下一阶段

## 8.3 `bootstrap.js` 只做 orchestration
当前 `scene/core/bootstrap.js` 已经是 orchestration layer，后续应该继续收缩，而不是继续往里面塞所有业务逻辑。

推荐：
- 门诊 visit 逻辑放进新的 runtime/store 模块
- doctor dialogue 单独拆模块
- 服务节点交互单独拆模块

## 8.4 queue / npc / dialogue 保持独立模块
前端扩展时必须保持这三类能力分离：
- `queue`
- `npc`
- `dialogue`

否则后续多人协作时会再次回到“大文件互相打补丁”的状态。

## 九、测试与验收建议
门诊系统开发不能只看页面效果，必须有明确的回归与验收基线。

## 9.1 API 测试
覆盖：
- `visit` 创建
- triage session 与 follow-up
- doctor consultation 消息往返
- 服务节点状态推进
- queue / patient / visit 查询

## 9.2 状态机测试
覆盖：
- visit lifecycle 合法迁移
- triage dialogue state 合法迁移
- doctor consultation 子状态机合法迁移
- 非法事件是否被正确拒绝

## 9.3 主流程回归测试
至少覆盖：
- 标准开药路径
- 检验回诊路径

## 9.4 记忆边界测试
覆盖：
- triage private memory 不泄漏到 doctor private memory
- shared memory 能正确承接已确认事实
- working state 不被误写成长期事实

## 9.5 前后端一致性测试
覆盖：
- 前端看到的门诊阶段与后端一致
- 场景节点能否正确触发后端状态变化
- 队列板、任务板、对话页是否同步更新

## 9.6 fallback 测试
覆盖：
- triage LLM 不可用时是否仍可 fallback
- doctor agent 不可用时系统是否仍可退化推进基本流程

## 十、协作红线
以下做法必须明确禁止：

1. **不把新 Agent 直接塞进 `server.py`**
   - 所有新 Agent 必须走标准 agent package 结构

2. **不把 EventBus 当决策器**
   - EventBus 只传播已发生事实，不决定主流程

3. **不把所有 Agent 的记忆混在一起**
   - shared / private / working 的边界必须保持

4. **不让前端退回到单大文件承载所有逻辑**
   - 新能力必须拆到对应模块

5. **不在主流程未稳定前优先做 supporting NPC 的复杂智能**
   - 先闭环，再增强氛围

---

## 十一、推荐执行原则
为了确保项目能真实推进，建议团队始终遵循以下原则：
- 先做主流程，再做复杂智能
- 先做状态机，再做 prompt
- 先保证流程可复现，再追求角色数量和沉浸感
- 先保证模块边界清晰，再扩大功能面

一句话总结：

**当前框架最适合的推进方式，不是“大而全扩展”，而是围绕 Visit 主线，逐步把 triage、doctor、service nodes 和环境角色收编进一个清晰、闭环、可持续维护的门诊系统。**
