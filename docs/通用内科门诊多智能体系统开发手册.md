# 通用内科门诊多智能体系统开发手册

## 1. 文档目标
本手册用于指导开发者在当前项目架构下，正确、可并行地开发一个新的对话 Agent，并完成最终实装。

这份手册重点回答四个问题：
1. 新 Agent 应该接入哪些后端与前端模块。
2. 多人并行开发时如何拆分任务，避免冲突与返工。
3. 各模块的边界是什么（状态机、事件、记忆、仓储）。
4. 如何用最小路径验证“可用”，并完成集成验收。

适用对象：
- 后端开发（Agent、API、状态机、事件、Repository）
- 前端开发（scene 交互、agent client、对话 UI）
- 测试与联调同学

## 3. 新 Agent 落地总原则

### 3.1 先状态机，后 Prompt
必须先定义：
1. 业务状态边界（Patient/Visit）
2. 对话子状态机（Agent Dialogue）

然后再设计 prompt、rules、graph。

### 3.2 先契约，后实现
先固定请求/响应 schema，再写 service/graph。避免前后端联调时字段漂移。

### 3.3 先测试，后前端
至少先有状态机测试和 API 流程测试，再接入前端 UI。

### 3.4 EventBus 只做广播，不做决策
状态迁移由状态机决定；EventBus 只在“状态已提交”后触发联动副作用。

### 3.5 记忆分层必须严格隔离
1. shared memory：跨 Agent 的稳定事实。
2. agent private memory：某个 Agent 私有过程状态。
3. working state：图运行时临时状态。

## 4. 并行开发模式（推荐三组并行）

为减少冲突，按纵切面并行，不按“每人改一点”横切。

### 4.1 A 组：状态与契约组（先行）
职责：定义“这个 Agent 是什么”。

输出物：
1. `backend/app/schemas/common.py` 中新增或复用 Agent 对话状态枚举。
2. `backend/app/agents/<new_agent>/state_machine.py`：完整转移表。
3. `backend/app/agents/<new_agent>/schemas.py`：Create/Message/Response 契约。
4. `backend/tests/test_state_machines.py`：happy path + invalid transition。

完成标准：
- 所有状态名、事件名可被 B 组与 C 组直接引用。

### 4.2 B 组：运行时组（依赖 A 组）
职责：让 Agent 真正可运行。

输出物：
1. `backend/app/agents/<new_agent>/service.py`
2. `backend/app/agents/<new_agent>/graph.py`
3. `backend/app/agents/<new_agent>/rules.py`
4. `backend/app/agents/<new_agent>/prompts.py`
5. `backend/app/agents/<new_agent>/state.py`
6. `backend/app/agents/<new_agent>/__init__.py`（工厂函数）

完成标准：
- `create_session` 与 `continue_session` 均可跑通。
- 能正确读写 shared/private memory。
- 能在关键里程碑发布事件。

### 4.3 C 组：接线与交互组（依赖 A/B）
职责：让系统能调用到新 Agent。

输出物：
1. `backend/app/api/routes/<new_agent>.py`
2. `backend/app/main.py`：注册 service、router、必要订阅。
3. `scene/agent/client.js`：新增 API 方法。
4. `scene/core/bootstrap.js`：新增发起会话与追问入口。
5. 对应 UI 模块：渲染对话与状态。

完成标准：
- 前端可创建会话、发送消息、获取最新对话。
- 失败态有明确提示，不会把 UI 卡死。

### 4.4 并行依赖关系
1. A 完成前：B/C 可以搭框架，但不能冻结实现。
2. B 完成后：C 可稳定接 API。
3. 集成时以 A 的 schema/state 命名为唯一基准。

## 5. 新 Agent 最小模板（可直接复制）

### 5.1 后端目录模板

```text
backend/app/agents/<new_agent>/
  __init__.py
  graph.py
  state.py
  state_machine.py
  schemas.py
  prompts.py
  rules.py
  service.py
```

### 5.2 最小职责模板
1. `state_machine.py`
   - 定义 `<NEW_AGENT>_TRANSITIONS`
   - 定义 `<NewAgent>DialogueStateMachine.transition`

2. `schemas.py`
   - `Create<NewAgent>SessionRequest`
   - `<NewAgent>MessageRequest`
   - `<NewAgent>SessionResponse`

3. `service.py`
   - `create_session(payload)`
   - `continue_session(session_id, payload)`
   - `prepare_context(...)`
   - `evaluate(...)`
   - `persist_result(...)`
   - `build_response(...)`

4. `graph.py`
   - `load_context -> evaluate -> persist -> build_response`
   - LangGraph 不可用时保留顺序 fallback。

5. `__init__.py`
   - `create_<new_agent>_service(...)`

### 5.3 API 路由模板
建议至少包含：
1. `POST /api/v1/<new-agent>-sessions`
2. `POST /api/v1/<new-agent>-sessions/{session_id}/messages`
3. `GET /api/v1/<new-agent>-sessions/{session_id}`

## 6. 最终实装流程（从零到可用）

### Step 0：定义边界
明确新 Agent 的输入、输出、与上游/下游阶段关系。

示例：
- 上游：`visit=in_consultation`
- 下游：`consultation_completed -> visit=waiting_payment`

### Step 1：补状态机与契约
完成 A 组输出，先让状态图和字段稳定。

### Step 2：实现 Agent Runtime
完成 B 组输出，跑通 service + graph。

### Step 3：接入 API 与容器
完成：
1. 路由文件
2. `main.py` 中 service 创建与 router 注册
3. 必要事件订阅

### Step 4：补前端调用链
完成：
1. `scene/agent/client.js` 新增 API 调用方法
2. `scene/core/bootstrap.js` 新增 create/message 流程
3. 对话 UI 模块渲染状态、消息、证据

### Step 5：联调与回归
先跑后端测试，再跑前后端联调。

### Step 6：验收与发布
按第 10 节验收清单逐条确认。

## 7. 后端集成锚点清单

### 7.1 容器装配
在 `backend/app/main.py`：
1. 创建新 Agent service
2. 注入 `app.state.container`
3. 注册 router
4. 订阅关键事件（如需）

### 7.2 事件类型
在 `backend/app/events/types.py` 中新增该 Agent 的关键事件常量，并保持命名统一。

### 7.3 状态推进
确保 `persist_result` 中：
1. 先完成状态迁移与持久化
2. 再 `bus.publish(...)`

若顺序反了，订阅侧会读取到不一致状态。

---

## 8. 前端集成锚点清单

### 8.1 API 客户端
在 `scene/agent/client.js` 新增：
1. `create<NewAgent>Session(payload)`
2. `send<NewAgent>Message(sessionId, payload)`
3. `get<NewAgent>Session(sessionId)`

### 8.2 Bootstrap 编排
在 `scene/core/bootstrap.js` 新增：
1. 发起会话动作（通常是 E 键触发后的某个分支）
2. 继续追问动作
3. 轮询后对话同步逻辑

### 8.3 UI 显示
保证以下行为：
1. `awaiting_patient_reply` 时可继续输入。
2. `completed` 时禁用发送。
3. `failed` 时可重试并提示错误信息。

## 9. 测试与验证策略

### 9.1 必测项
1. 状态机：合法迁移、非法迁移。
2. Service：create/continue 路径。
3. API：完整端到端流程。
4. Repository：新增持久化逻辑时补测试。
5. 多 Agent 场景：private memory 不串写。

### 9.2 建议命令

```powershell
Set-Location backend
python -m pytest tests/test_state_machines.py -v
python -m pytest tests/test_api_flow.py -v
```

如只验证新 Agent，可使用 `-k` 过滤。

### 9.3 最小跑通路径
1. 创建 visit。
2. 推进到新 Agent 的准入状态。
3. 创建新 Agent session。
4. 发送至少 1 轮追问回复。
5. 返回有效 `patient + dialogue + visit_state`。

## 10. 并行集成验收清单（Definition of Done）

### 10.1 契约与状态
- [ ] 对话状态枚举稳定，命名已冻结。
- [ ] 状态机测试通过。
- [ ] API schema 文档化且字段完整。

### 10.2 运行时
- [ ] create/continue 会话均可执行。
- [ ] shared/private/working 三层数据边界清晰。
- [ ] 事件发布时机正确（状态提交后）。

### 10.3 前后端接线
- [ ] 前端能发起新会话。
- [ ] 前端能继续多轮对话。
- [ ] 轮询与详情接口能稳定刷新 UI。

### 10.4 质量与风险
- [ ] `test_state_machines` 与 `test_api_flow` 无回归。
- [ ] 无跨 Agent 私有记忆污染。
- [ ] 失败态有 fallback，不会中断主流程。

## 11. 常见问题与规避

### 11.1 在 `server.py` 或单文件里直接堆逻辑
后果：不可维护、难以并行开发。
规避：严格按 `agents/<new_agent>/` 包结构实现。

### 11.2 让 EventBus 决定流程
后果：状态不可追踪，链路不稳定。
规避：EventBus 只广播，状态机做决策。

### 11.3 单一 `session_id` 误当多 Agent 真相
后果：上下文混淆。
规避：按 `agent_type` 隔离 private memory，并在患者视图中明确 active agent。

### 11.4 前端绕过状态直接推进
后果：前后端状态不一致。
规避：所有推进动作由后端状态机裁决，前端只展示与触发。

## 12. 推荐执行顺序（3 天并行版）

### Day 1
1. A 组冻结状态机与 schema。
2. B 组搭 service/graph 骨架。
3. C 组预接前端按钮与 client 方法签名。

### Day 2
1. B 组完成运行时逻辑并通过后端测试。
2. C 组完成 API 接线与对话 UI 联调。

### Day 3
1. 全链路回归。
2. 修复边界问题。
3. 按第 10 节打勾后合并。
