## 0 · 约定与术语

| 术语 | 含义 |
|---|---|
| 主组 | A、B、C、D 四个开发小组 |
| 全院总线 | 由教学组提供的 Redis Pub/Sub 实例，所有跨组事件均通过此发布与订阅 |
| LLM 网关 | 由教学组提供的统一大模型调用入口；所有组的 LLM 调用必须经此发出 |
| 就诊（Encounter） | 患者一次完整的医院就诊过程，从挂号开始，到出院或主动终止结束 |
| 跨组事件 | 涉及两个及以上小组之间数据流动的事件（如急诊→ICU 入院） |
| 内部事件 | 仅在本组进程内流转、不需要其他组消费的事件 |

**全文中"工作日"指周一至周五，跨度按校历计算。**

## 1 · 标识符（ID）体系

### 1.1 患者标识 Patient ID

**格式：** `P-{8 位小写十六进制}`

**示例：** `P-a1b2c3d4`

**生成规则：** 由首次接收患者的小组生成。建议使用 `uuid4` 取前 8 位 hex；不得使用自增序列（避免跨组冲突）。

**唯一性约束：** 必须在全院范围内唯一；同一自然人在不同就诊中应使用同一 Patient ID（如系统未保存历史档案，可视为新患者）。

### 1.2 就诊标识 Encounter ID

**格式：** `E-{14 位时间戳}-{4 位小写十六进制}`

时间戳为 `YYYYMMDDHHmmss`，使用东八区本地时间。

**示例：** `E-20260507143022-1a2b`

**生成规则：** 由发起就诊的小组生成（通常为挂号 / 分诊环节）。同一患者多次就诊须分配不同 Encounter ID。

**有效期：** 一次就诊从挂号到出院 / 终止之间使用同一 Encounter ID，**不得**在转诊时重新生成。

### 1.3 床位与房间标识

| 类型 | 格式 | 示例 |
|---|---|---|
| 床位 Bed ID | `B-{病区代码}-{两位床号}` | `B-ICU01-12`、`B-WARD03-07` |
| 房间 Room ID | `R-{科室代码}-{两位房间号}` | `R-ED-04`、`R-OUT-12` |

科室代码采用三字母大写缩写：`ED`（急诊）、`ICU`、`OUT`（门诊）、`WARD`（住院）、`LAB`、`IMG`（影像）、`PHA`（药房）。

### 1.4 检验、影像、医嘱标识

| 类型 | 前缀 | 示例 |
|---|---|---|
| 检验单 | `LAB-` | `LAB-20260507-00031` |
| 影像单 | `IMG-` | `IMG-20260507-00012` |
| 医嘱 | `ORD-` | `ORD-20260507-00045` |
| 干预记录 | `INT-` | `INT-20260507-00007` |

后缀为 `日期 + 5 位序号`，由发起方在本日内自增。

### 1.5 Agent 实例标识

**格式：** `{角色}-{2 位序号}`

**示例：** `triage-01`、`doctor-internal-02`、`bedside-monitor-03`

**约束：** Agent ID 在本组内唯一；跨组事件须额外携带 `producer` 字段（见 §4.3）以区分小组。

### 1.6 ID 唯一性的冲突处理

任一接收方收到与本地已有记录冲突的 ID 时，**必须**以"先入库者为准"，并向发送方返回 `409 Conflict`，由发送方重新生成 ID。

## 2 · 患者就诊状态字典

全院统一使用以下 12 个状态。**各组内部允许保留更细粒度的子状态，但对外暴露的状态字段必须落入下表。**

### 2.1 主流程状态（9 项）

| 状态码 | 含义 | 典型触发方 |
|---|---|---|
| `ARRIVED` | 患者到达医院，尚未挂号 | A 组（大厅入口） |
| `REGISTERED` | 挂号完成 | A 组 |
| `TRIAGED` | 分诊完成，等待就诊 | A / B 组 |
| `IN_CONSULTATION` | 就诊中（医生面诊） | A / B 组 |
| `IN_EXAM` | 检查中（检验、影像） | B 组 |
| `IN_TREATMENT` | 治疗中（处置、给药） | B / C / D 组 |
| `ADMITTED` | 已收入院 | C / D 组 |
| `DISCHARGED` | 出院 | D 组 |
| `COMPLETED` | 流程闭环（含随访结束） | D 组 |

### 2.2 辅助状态（3 项）

| 状态码 | 含义 |
|---|---|
| `TRANSFERRING` | 跨科 / 跨组转诊中（短暂状态，超时 5 分钟自动回滚） |
| `CANCELLED` | 患者主动放弃就诊 |
| `ERROR` | 出现异常需人工介入；记录后必须立即向总线发 `alert.raised` 事件 |

### 2.3 合法状态迁移图

```
                ARRIVED
                   ↓
                REGISTERED
                   ↓
                 TRIAGED
                   ↓
            IN_CONSULTATION ──┐
                ↑    ↓        │
              IN_EXAM         │
                ↑    ↓        │
            IN_TREATMENT      │
                ↓             │
              ADMITTED ◀──────┘
                ↓
              DISCHARGED
                ↓
              COMPLETED

任意状态 → TRANSFERRING（短暂）→ 目标状态
任意状态 → CANCELLED / ERROR（终止）
```

不合法迁移（如 `ARRIVED → ADMITTED` 跳过中间环节）**必须**返回 `422 Unprocessable Entity`。

### 2.4 各组内部状态映射示例

各组**必须**在本组 README 中提供一份内部状态到全局状态的映射表。下面是已知的几组对应：

| 组 | 内部状态 | → 全局状态 |
|---|---|---|
| A | `triaged` | `TRIAGED` |
| A | `in_consultation` | `IN_CONSULTATION` |
| B | `TRIAGE_COMPLETE` | `TRIAGED` |
| B | `WAITING_FOR_PHYSICIAN` | `TRIAGED` |
| B | `UNDER_EVALUATION` | `IN_CONSULTATION` |
| C | `care_phase=stable` | `ADMITTED`（active 状态下） |
| D | `phase=ADMISSION` | `ADMITTED` |

## 3 · 分诊等级与分区

### 3.1 CTAS 五级

全班统一采用 **CTAS（Canadian Triage and Acuity Scale）五级**作为对外分诊等级。各组内部如使用其他标准（ESI、A-D 等），**必须**在对外发布事件时映射回 CTAS。

| 级别 | 含义 | 中文 | 最长候诊时间 |
|---|---|---|---|
| `L1` | Resuscitation | 危急（即刻处置） | 0 分钟 |
| `L2` | Emergent | 紧急 | ≤ 15 分钟 |
| `L3` | Urgent | 较急 | ≤ 30 分钟 |
| `L4` | Less Urgent | 次急 | ≤ 60 分钟 |
| `L5` | Non-urgent | 非急 | ≤ 120 分钟 |

### 3.2 zone 派生

`zone` 字段由 CTAS 等级**自动派生**，不允许独立设置：

| CTAS | zone |
|---|---|
| L1, L2 | `red` |
| L3 | `yellow` |
| L4, L5 | `green` |

### 3.3 升降级与 hooks

CTAS 等级可在就诊过程中调整。每次调整**必须**伴随一条 `triage.upgraded` 或 `triage.downgraded` 事件，并在 `data.reason` 中给出原因。

常用 hooks（升级触发条件，仅供参考，各组可扩展）：

- `abnormal_vitals` — 任一生命体征跌出安全区间
- `deterioration` — 短时间内连续两次评估恶化
- `lab_critical` — 出现危急值
- `green_channel` — 绿色通道病种（STEMI、卒中、严重创伤、脓毒症）

## 4 · 事件总线与消息协议

### 4.1 总线选型

**全院总线采用 Redis Pub/Sub。** 教学组统一部署一个 Redis 实例，连接信息通过环境变量下发：

```bash
HOSPITAL_REDIS_HOST=hospital-redis
HOSPITAL_REDIS_PORT=6379
HOSPITAL_REDIS_DB=0
HOSPITAL_REDIS_PASSWORD=<由教学组分发>
```

不允许各组自建 Redis；不允许通过其他通道（HTTP、文件、数据库）替代事件总线传递跨组事件。

### 4.2 频道命名

频道命名采用三段式：

```
hospital.<domain>.<event>
```

`<domain>` 取值：`patient`、`encounter`、`triage`、`order`、`vital`、`exam`、`alert`、`transfer`、`agent`。

**示例：**

```
hospital.patient.registered
hospital.patient.triaged
hospital.transfer.requested
hospital.vital.recorded
hospital.alert.raised
```

订阅时**应当**使用具体频道；如需聚合订阅，可使用 `PSUBSCRIBE hospital.patient.*`。

### 4.3 消息信封 (Envelope) Schema

**所有发布到总线的消息必须使用以下信封结构：**

```json
{
  "event_id": "evt_01HX7K3M2N4PQRSTVWXYZA1B2C",
  "event_type": "patient.triaged",
  "schema_version": "1.0",
  "occurred_at": "2026-05-07T14:30:22.123+08:00",
  "producer": "groupB.ed",
  "patient_id": "P-a1b2c3d4",
  "encounter_id": "E-20260507143022-1a2b",
  "correlation_id": "evt_01HX7K3M2N4PQRSTVWXYZA1B2C",
  "data": { }
}
```

字段说明：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `event_id` | string | 是 | 全局唯一事件 ID。建议使用 ULID（带前缀 `evt_`） |
| `event_type` | string | 是 | 与频道 `<domain>.<event>` 部分一致 |
| `schema_version` | string | 是 | 当前固定为 `"1.0"` |
| `occurred_at` | string (ISO 8601) | 是 | 事件实际发生时间，**必须**带时区 |
| `producer` | string | 是 | 形如 `groupX.<service>`，例如 `groupA.outpatient`、`groupC.icu` |
| `patient_id` | string | 是 | 关联患者 ID |
| `encounter_id` | string | 否 | 若与具体就诊相关则必填 |
| `correlation_id` | string | 否 | 用于关联同一业务流的多个事件，默认等于本事件 `event_id` |
| `data` | object | 是 | 事件具体载荷，由各 `event_type` 各自定义（见附录 A） |

### 4.4 标准事件类型清单

下表为 v1.0 必须支持的事件类型。各组按角色发布 / 订阅，对应关系详见 §9。

| event_type | 含义 | 主要发布方 | 主要订阅方 |
|---|---|---|---|
| `patient.arrived` | 患者到达医院 | A | A |
| `patient.registered` | 挂号完成 | A | A、B |
| `patient.triaged` | 分诊完成 | A、B | A、B、C |
| `patient.routed` | 分诊后被路由至某科室 | A、B | 目标科室所在组 |
| `patient.transferred` | 跨组转诊已生效 | 源组 | A（用于前端显示）、目标组 |
| `patient.admitted` | 入院完成 | C、D | A、D |
| `patient.discharged` | 出院 | D | A、C |
| `encounter.opened` | 一次就诊开始 | A、B | A |
| `encounter.closed` | 一次就诊结束 | D | A |
| `triage.upgraded` | 分诊等级上调 | 任一 | 全员 |
| `triage.downgraded` | 分诊等级下调 | 任一 | 全员 |
| `vital.recorded` | 生命体征记录 | B、C | C（用于风险评估） |
| `vital.critical` | 出现危急值 | B、C | 全员 |
| `lab.ordered` | 检验申请 | B、C | B（检验科逻辑） |
| `lab.resulted` | 检验回报 | B | 申请方 |
| `imaging.ordered` | 影像申请 | B、C | B |
| `imaging.resulted` | 影像回报 | B | 申请方 |
| `order.issued` | 医嘱下达 | B、C、D | 执行方 |
| `order.executed` | 医嘱执行完成 | 执行方 | 申请方 |
| `alert.raised` | 告警产生 | 任一 | A（前端显示）、教学组 |
| `alert.acknowledged` | 告警被确认 | 处理方 | A |

事件清单**可以**在 v1.x 内补充扩展；**不得**删除或修改既有事件的字段语义。

### 4.5 可靠性

Redis Pub/Sub 默认为 At-most-once 投递。为保证跨组关键事件不丢失：

- 关键事件（标记为 ★ 的事件类型）**必须**同时写入 Redis Stream `hospital:journal`（持久日志），消费方在重启后可通过 `XRANGE` 回放。
- 标记为 ★ 的事件类型：`patient.transferred`、`patient.admitted`、`patient.discharged`、`vital.critical`、`alert.raised`。
- 非关键事件**可以**只走 Pub/Sub。

### 4.6 D 组多进程桥接要求

D 组当前的 `multiprocessing.Queue` 内部消息**不得**直接跨进程传递给其他组。**必须**满足以下条件：

1. 在 dispatcher 外部封装一个 HTTP / Redis 桥接服务，端口由教学组分配；
2. 内部 `Message` 数据类的关键事件（入院、出院、告警）**必须**在桥接层翻译为标准信封并发布到 Redis；
3. 反方向：从 Redis 订阅到的相关事件**必须**翻译为内部 `Message` 投递到对应进程。

桥接最小骨架参考代码见 §附录 C。

## 5 · HTTP API 规范

### 5.1 基础路径

所有对外 HTTP 接口**必须**使用以下前缀：

```
http://<group-host>:<port>/api/v1/
```

`v1` 与本契约 v1.x 共生命周期；进入 v2 须教学组评审通过。

### 5.2 请求 / 响应格式

- 请求与响应**必须**使用 `application/json; charset=utf-8`。
- 字段命名**必须**使用 `snake_case`（与本契约保持一致）。
- 时间戳**必须**为 ISO 8601 字符串，带时区。
- 数值字段**不得**使用字符串包裹（如 `"age": "23"` 不合规）。

**通用响应外壳：**

```json
{
  "ok": true,
  "data": { },
  "error": null,
  "trace_id": "trc_01HX..."
}
```

错误时：

```json
{
  "ok": false,
  "data": null,
  "error": {
    "code": "STATE_TRANSITION_INVALID",
    "message": "Cannot transition from ARRIVED to ADMITTED",
    "details": { }
  },
  "trace_id": "trc_01HX..."
}
```

### 5.3 错误码体系

错误码采用大写下划线格式，命名空间 `<domain>_<reason>`：

| 错误码 | HTTP 状态 | 含义 |
|---|---|---|
| `ID_MALFORMED` | 400 | ID 格式不符合 §1 规范 |
| `STATE_TRANSITION_INVALID` | 422 | 状态迁移不合法 |
| `TRIAGE_LEVEL_INVALID` | 422 | 不在 L1–L5 范围内 |
| `ENCOUNTER_NOT_FOUND` | 404 | 就诊记录不存在 |
| `PATIENT_CONFLICT` | 409 | Patient ID 与已有记录冲突 |
| `LLM_GATEWAY_UNAVAILABLE` | 503 | LLM 网关不可用 |
| `LLM_RATE_LIMITED` | 429 | 触发 LLM 限流 |
| `EVENT_SCHEMA_INVALID` | 400 | 事件信封不符合 §4.3 |
| `INTERNAL_ERROR` | 500 | 未分类错误，须同时上报 `alert.raised` |

### 5.4 幂等性

所有写操作（POST、PUT、DELETE）**应当**支持幂等：

- 客户端**应当**在请求头中携带 `Idempotency-Key: <client-generated-uuid>`；
- 服务端在 24 小时内对相同 key 的相同请求**必须**返回相同结果。

## 6 · 跨组转诊接口

跨组转诊是本契约的核心接口，**所有四个组必须实现**。

### 6.1 发起转诊

**请求**

```
POST /api/v1/encounters/{encounter_id}/transfer
Content-Type: application/json
Idempotency-Key: <uuid>

{
  "from_group": "groupB.ed",
  "to_group": "groupC.icu",
  "reason": "STEMI 患者经 PCI 后需进入 ICU 监护",
  "ctas_level": "L1",
  "summary": {
    "chief_complaint": "持续胸痛 2 小时",
    "key_findings": ["ST 段抬高 V1-V4", "肌钙蛋白阳性"],
    "active_diagnoses": ["STEMI（前壁）"],
    "active_orders": ["ORD-20260507-00045"],
    "vital_baseline": {
      "hr": 88, "sbp": 110, "dbp": 70, "spo2": 96, "temp": 36.8, "rr": 18
    }
  },
  "requested_resources": {
    "bed_type": "ICU",
    "monitor": true,
    "ventilator_standby": true
  }
}
```

**响应**

```json
{
  "ok": true,
  "data": {
    "transfer_id": "TRF-20260507-00003",
    "status": "accepted",
    "assigned_bed": "B-ICU01-12",
    "expected_eta_minutes": 5
  },
  "error": null,
  "trace_id": "trc_..."
}
```

接收方**必须**在 5 秒内返回 `accepted`、`rejected` 或 `pending`。`pending` 应当附带 `retry_after_seconds`。

### 6.2 转诊事件流

发起方在收到 `accepted` 后**必须**：

1. 将本组内患者状态置为 `TRANSFERRING`；
2. 向总线发布 `patient.transferred` 事件；
3. 在患者实际抵达接收方后，发起方将状态置为 `COMPLETED`（本组流程结束），接收方将状态置为 `IN_CONSULTATION` 或 `ADMITTED`（视情况）。

## 7 · LLM 网关

### 7.1 统一接入

所有大模型调用**必须**经由教学组提供的 LLM 网关：

```
POST http://hospital-llm-gateway:8080/v1/chat/completions
```

接口与 OpenAI Chat Completions 兼容。各组**不得**直接调用任何外部 LLM 服务（OpenAI、Azure、Dashscope、DeepSeek 等）。

### 7.2 模型白名单

| 模型 ID | 用途 | 备注 |
|---|---|---|
| `gpt-4o` | 复杂推理（医生主诊、跨学科会诊） | 优先 |
| `gpt-4o-mini` | 一般对话、分诊问询 | 默认 |
| `qwen-max` | 国产备选 | 网关自动 fallback |
| `qwen-vl-max` | 多模态（图像分诊） | 仅 ayunsis 类场景 |
| `deepseek-chat` | 成本敏感的批量任务 | 仿真 / 测试用 |

请求 body 中的 `model` 字段**必须**为上表 ID；其他模型由网关返回 `400 INVALID_MODEL`。

### 7.3 凭据管理

网关凭据通过环境变量下发：

```bash
HOSPITAL_LLM_GATEWAY_URL=http://hospital-llm-gateway:8080
HOSPITAL_LLM_API_KEY=<由教学组分发>
```

代码中**不得**硬编码 API key；**不得**在 README、提交记录、issue 中粘贴 API key。一经发现，全组本周成绩按零分计算。

### 7.4 速率与缓存

- 每组每分钟限额 60 次调用；超限网关返回 `429 LLM_RATE_LIMITED`，调用方**应当**指数退避重试。
- 网关默认对 prompt 完全相同的请求返回缓存结果（TTL 5 分钟）。需要绕过缓存的请求**必须**在 header 中显式声明 `X-Cache: bypass`。

### 7.5 安全护栏

LLM 输出**必须**满足以下安全策略：

- **规则下限**：分诊、用药、手术指征类决策**必须**先经规则引擎判定；LLM 仅在规则结果之上做"升级"，不得"降级"。例如规则判定 L2 时，LLM 不得改判为 L4。
- **禁止内容**：明确诊断陈述（如"你得了肺癌"）、治疗承诺（如"吃这个药一定好"）。LLM 输出中检出此类表达时，调用方**必须**截留并向总线发 `alert.raised` 事件。
- **结构化输出**：医生 / 护士 Agent 的结论性输出**必须**使用 JSON Schema 约束（推荐 OpenAI Structured Outputs / function calling）。

## 附录 C · Python 参考实现片段

### C.1 发布事件（任一组通用）

```python
import os, json, uuid, time, redis
from datetime import datetime, timezone, timedelta

CN = timezone(timedelta(hours=8))
r = redis.Redis(
    host=os.environ["HOSPITAL_REDIS_HOST"],
    port=int(os.environ["HOSPITAL_REDIS_PORT"]),
    password=os.environ.get("HOSPITAL_REDIS_PASSWORD"),
    decode_responses=True,
)

def publish_event(event_type: str, patient_id: str, data: dict,
                  encounter_id: str | None = None,
                  producer: str = "groupA.outpatient",
                  correlation_id: str | None = None,
                  durable: bool = False):
    eid = "evt_" + uuid.uuid4().hex[:26].upper()
    envelope = {
        "event_id": eid,
        "event_type": event_type,
        "schema_version": "1.0",
        "occurred_at": datetime.now(CN).isoformat(timespec="milliseconds"),
        "producer": producer,
        "patient_id": patient_id,
        "encounter_id": encounter_id,
        "correlation_id": correlation_id or eid,
        "data": data,
    }
    payload = json.dumps(envelope, ensure_ascii=False)
    channel = f"hospital.{event_type}"
    r.publish(channel, payload)
    if durable:
        r.xadd("hospital:journal", {"payload": payload}, maxlen=100_000)
    return eid
```

### C.2 订阅事件

```python
def subscribe(patterns: list[str], handler):
    pubsub = r.pubsub()
    for p in patterns:
        pubsub.psubscribe(p)
    for msg in pubsub.listen():
        if msg["type"] != "pmessage":
            continue
        try:
            envelope = json.loads(msg["data"])
            handler(envelope)
        except Exception as e:
            # 必须发 alert.raised，不允许吞异常
            publish_event("alert.raised",
                          patient_id="P-00000000",
                          data={"source": "subscribe", "error": str(e)})
```

### C.3 调用 LLM 网关

```python
import httpx

def llm_chat(messages, model="gpt-4o-mini", **kwargs):
    url = os.environ["HOSPITAL_LLM_GATEWAY_URL"] + "/v1/chat/completions"
    headers = {"Authorization": f"Bearer {os.environ['HOSPITAL_LLM_API_KEY']}"}
    payload = {"model": model, "messages": messages, **kwargs}
    resp = httpx.post(url, json=payload, headers=headers, timeout=30)
    if resp.status_code == 429:
        raise RuntimeError("LLM_RATE_LIMITED")
    resp.raise_for_status()
    return resp.json()
```

### C.4 D 组桥接服务最小骨架

```python
# bridge.py — 在 D 组 dispatcher 旁运行
from fastapi import FastAPI
import threading

app = FastAPI()

# Redis → multiprocessing
def from_redis_to_dispatcher():
    def handler(envelope):
        if envelope["event_type"] == "patient.admitted":
            dispatcher.inbox.put(Message(
                priority=1, sender_id="bridge",
                receiver_id="NURSE_01",
                msg_type="ADMISSION_REQUEST",
                content=envelope["data"],
            ))
    subscribe(["hospital.patient.admitted"], handler)

# multiprocessing → Redis
@app.post("/internal/event")
def internal_event(payload: dict):
    publish_event(
        event_type=payload["event_type"],
        patient_id=payload["patient_id"],
        data=payload["data"],
        producer="groupD.inpatient",
        durable=payload.get("durable", False),
    )
    return {"ok": True}

threading.Thread(target=from_redis_to_dispatcher, daemon=True).start()
```
